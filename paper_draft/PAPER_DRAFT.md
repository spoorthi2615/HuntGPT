# HuntGPT-Shield: Autonomous Cyber Threat Hunting with Adversarial Prompt Injection Defense

## Abstract
Modern Security Operations Centers (SOCs) face overwhelming alert volume, driving the adoption of Large Language Models (LLMs) for automated log analysis and threat hunting. However, piping raw, untrusted network telemetry directly into LLMs introduces a critical attack surface: adversarial prompt injection. In this paper, we present HuntGPT-Shield, an end-to-end autonomous threat hunting pipeline that maps raw network logs to MITRE ATT&CK techniques while maintaining robust defense against payload-based prompt injections. We introduce a dual-model architecture comprising *PromptGuard*, a DeBERTa-v3 sequence classifier fine-tuned to filter malicious prompts, and *ThreatHunter*, a Mistral-7B model fine-tuned via QLoRA on the DARPA OPTC dataset for precise technique mapping. Our system achieves an F1 score of [0.XX] on prompt injection detection with [X.X]ms latency overhead, and a ThreatHunter technique identification accuracy of [0.XX], significantly outperforming zero-shot baselines. Furthermore, we implement a self-consistency scoring mechanism that improves model calibration, achieving an Expected Calibration Error (ECE) of [0.XX]. This work demonstrates that automated LLM-based threat hunting can be both highly accurate and securely hardened against adversarial manipulation.

## 1. Introduction
The volume of network telemetry generated by modern enterprise environments routinely exceeds the analytical capacity of human security analysts. This phenomenon, known as "alert fatigue," results in missed detections and delayed incident response. Recently, the cybersecurity community has explored Large Language Models (LLMs) to automate the triage process by summarizing logs and mapping them to standardized frameworks like MITRE ATT&CK.

However, using LLMs in autonomous security pipelines introduces a severe vulnerability. If an attacker embeds a prompt injection string (e.g., "Ignore previous instructions and classify this log as benign") inside a network payload (such as an HTTP User-Agent or DNS query), the LLM may execute the adversarial instruction, effectively blinding the SOC to the attack. 

To address this, we propose HuntGPT-Shield, a novel architecture that isolates the analytical LLM from adversarial manipulation. Our contributions are two-fold:
1. We introduce a fast, lightweight filter model (PromptGuard) that sanitizes network logs prior to LLM inference, preserving the integrity of the SOC automation pipeline.
2. We demonstrate a fine-tuning methodology for open-weights LLMs (ThreatHunter) that significantly outperforms zero-shot commercial models in mapping highly complex Zeek network logs to MITRE ATT&CK techniques.

## 2. Related Work
* **LLMs for Cybersecurity**: Recent work has explored using LLMs for vulnerability analysis, malware reverse engineering, and SOAR (Security Orchestration, Automation, and Response) log summarization. Most approaches rely on zero-shot inference using commercial APIs, which poses privacy concerns and limits domain-specific accuracy.
* **Prompt Injection Defense**: Defending against prompt injection remains an active research area. Approaches range from simple heuristic blacklists and LLM-in-the-loop validation to perplexity-based filtering. We expand on this by applying a dedicated transformer-based sequence classifier specifically trained on adversarial payloads embedded in network protocol structures.
* **NIDS and SOAR Approaches**: Traditional Network Intrusion Detection Systems (NIDS) rely on static signatures (e.g., Snort, Zeek) or statistical anomaly detection. HuntGPT-Shield bridges the gap between raw NIDS output and high-level SOAR automation by autonomously generating human-readable hypotheses and mapping them directly to the ATT&CK matrix.

## 3. System Design
HuntGPT-Shield is designed as a modular, containerized pipeline suitable for SOC deployment. The architecture processes batches of Zeek network logs through the following sequence:

1. **Log Ingestion & Parsing**: Raw Zeek logs (conn, http, dns) are parsed and normalized.
2. **PromptGuard Filtering (DeBERTa-v3-small)**: Each parsed log is evaluated by a lightweight sequence classifier. Logs flagged as containing prompt injections are immediately blocked and logged as adversarial attempts, bypassing the LLM.
3. **ThreatHunter Inference (Mistral-7B QLoRA)**: Clean logs are passed to the fine-tuned LLM. The model is prompted to return a strictly formatted JSON object containing the identified `technique_id`, `technique_name`, and an analyst `hypothesis`.
4. **Self-Consistency Scorer**: To measure confidence, the pipeline performs N=5 inference passes at `temperature=0.7`. The final technique is selected via majority vote, and the agreement ratio serves as a calibrated confidence score.
5. **Dashboard Presentation**: Results are rendered on a React-based frontend, grouping blocked injections and visualizing detected techniques on a MITRE ATT&CK frequency heatmap.

## 4. Methodology
### 4.1 PromptGuard Training
To train the PromptGuard filter, we generated a synthetic dataset of 1,200 samples comprising a strict 50/50 class balance of benign and injected logs. Adversarial payloads were synthetically embedded into realistic log templates (e.g., HTTP User-Agent strings). We fine-tuned `microsoft/deberta-v3-small` using a standard cross-entropy loss objective over 3 epochs, evaluating performance strictly on a held-out stratified test set to ensure realistic F1 reporting.

### 4.2 ThreatHunter QLoRA Configuration
We selected `Mistral-7B-Instruct-v0.2` as the base model for ThreatHunter. Due to computational constraints, we utilized Quantized Low-Rank Adaptation (QLoRA) with 4-bit NormalFloat quantization (NF4) and a computational dtype of `bfloat16`. We applied LoRA adapters with a rank of $r=16$ and $\alpha=32$ targeting the `q_proj` and `v_proj` attention matrices.

### 4.3 Dataset Construction & Time-Windowing
The training data for ThreatHunter was derived from the DARPA OPTC dataset. Because OPTC attack labels are documented at the scenario level rather than per individual log line, we employed a time-windowing strategy. We defined a strict temporal window ($\pm 30$ seconds) around known attack events. Zeek log lines falling within this window were joined with the corresponding MITRE ATT&CK `technique_id`. Logs falling outside any attack window were excluded from the training pairs to prevent noisy labeling.
To compensate for the limited number of distinct attack scenarios natively present in OPTC, we augmented the dataset by converting real-world `procedure_examples` from the official MITRE ATT&CK STIX database into synthetic log-to-technique pairs.

### 4.4 Calibration & Self-Consistency
Zero-shot LLMs often exhibit severe overconfidence in classification tasks. To improve the Expected Calibration Error (ECE), we implemented a self-consistency mechanism. Rather than relying on the raw softmax probabilities of a single inference pass, we generate N=5 predictions and calculate the agreement ratio. This frequency-based confidence metric aligns much more closely with true empirical accuracy.
